## Overview
Regression is a browser roguelike built on PostgreSQL's own regression test suite. Each floor is a query; you survive by predicting its output. It's for engineers who want to sharpen SQL semantics into reflexes and enjoy the tension of a run they can lose. A serious correctness harness, turned into a dungeon.

## Problem
Most SQL practice is 'write a query to get X.' Far fewer tools test the harder skill: *reading* a gnarly query and knowing precisely what Postgres does — NULL propagation, type coercion, window frames, ordering guarantees, integer overflow errors. The pgrust/Postgres-regression-tests HN thread is a reminder that this corpus is a goldmine of edge cases nobody games with.

## How it works
A run is a descent of ~10 floors of increasing difficulty. Each floor shows a query and (when needed) the seed table state. You choose your prediction: the exact result set (multiple-choice among plausible near-misses) or, for error cases, the exact error class ('division by zero', 'invalid input syntax for integer'). Correct = descend and bank score; wrong = death, show the real output, post run length. **Relics** picked up along the way bend rules (a 'peek' that reveals column count, a 'revive' once per run). Difficulty tiers pull from progressively hairier test files. Daily seeded gauntlet + endless mode. Streaks and depth are the score.

## Technical approach
Stack: static site + a real Postgres to *generate* content offline (not at runtime). Data source: the `src/test/regress/sql` and expected `expected/*.out` files from the Postgres source tree. A build script runs each statement against a pinned Postgres in Docker, capturing exact rows/errors, then emits a JSON bank of `{setup, query, correctAnswer, distractors}`. Distractors are auto-generated by mutating the true output (drop a row, flip a NULL, change an error class) so wrong answers are seductive. Client is pure TS — no live DB, so it's cheap and offline. Key structures: a floor = a graded question object; run state = seeded RNG over difficulty-bucketed banks. Hard part: authoring good *distractors* (must be wrong-but-plausible, not obviously silly) and curating from thousands of regression statements down to ones that are fair to predict without the full schema in your head.

## v1 scope
- Offline-generated bank from 2–3 regression files (int8, case, window)
- 10-floor seeded daily run, multiple-choice answers
- Permadeath + depth score + local streak
- 'Reveal real output' on death

## Out of scope
- Live in-browser query execution
- Free-text answer grading
- Multiplayer/leaderboards, accounts
- Non-Postgres dialects

## Risks & unknowns
- Many regression queries need heavy schema context — filtering to self-contained ones is the real work.
- Multiple-choice may be too easy; free-text prediction is much harder to grade.
- Keeping it fun vs. feeling like a certification exam.

## Done means
A build step produces a versioned JSON question bank from real Postgres output; a player can start the daily run, answer floors, die on a wrong prediction, see the actual Postgres result, and get a shareable depth score — with the same seed yielding the same gauntlet across devices.
