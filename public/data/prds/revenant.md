## Overview
Revenant is a browser tower-defense whose enemies are generated by running a real mutation-testing tool against the player's own codebase. It turns the tedious, slow, guilt-inducing chore of "why is my mutation score so bad" into a wave-based siege where under-tested code is literally overrun. For developers who know mutation testing is good for them and never run it.

## Problem
Mutation testing (Stryker, infection, mutmut) is the gold standard for measuring whether your tests actually *assert* anything — but the reports are dense JSON, the runs are slow, and nobody stares at a 61% mutation-score number long enough to fix it. Line coverage lies; a surviving mutant is proof your test would pass even if the code were wrong. That signal deserves a face.

## How it works
Point Revenant at a repo. It runs the mutation tool, parses the report, and stages a level: each source file is a *lane*, each mutant is an enemy that spawns at the file and marches toward your `main` branch at the center. Your tests are turrets — a test placed over a lane kills (kills = detects) any mutant it covers. Killed mutants pop with a satisfying green burst. Surviving mutants become glowing **Revenants** that reach main and drain your health bar, each one annotated with its exact diff (`- if (x > 0)` → `+ if (x >= 0)`) and the file:line it haunts. Beat a level by raising your kill rate; the post-game screen is a punch-list of the specific assertions to add. A daily "raid" re-runs on your latest commit so regressions spawn fresh revenants.

## Technical approach
Stack: a small Node/TS runner wrapping StrykerJS (JS/TS), infection (PHP), or mutmut (Python), invoked with their JSON reporters. Parse the report into a `{file, mutant, status: killed|survived|no-coverage, diff, coveredByTests[]}` model. Frontend is a Canvas/Pixi.js tower-defense fed by that model — lanes = files sorted by mutant density, turret coverage derived from the test-to-mutant map the tools already emit. The genuinely hard part is *pacing*: mutation runs take minutes, so Revenant caches per-commit reports in SQLite and diffs mutant sets between commits to spawn only *new* revenants incrementally, keeping the daily raid fast. Balancing wave difficulty against real mutation counts (some files have 400 mutants) needs bucketing/sampling so a monster file doesn't produce an unplayable swarm.

## v1 scope
- One language (StrykerJS/TS only)
- Single static level from one mutation report, no live re-runs
- Turrets = tests, auto-placed from the coverage map; player only chooses which lanes to reinforce
- Health bar + kill-rate score + end-screen punch-list

## Out of scope
- Multiplayer / leaderboards across repos
- Multi-language support
- Real-time incremental raids (v2)

## Risks & unknowns
- Mutation runs may be too slow for a fun loop on big repos — mitigate with file allowlists and sampling.
- Mapping mutants→covering-tests reliably across tools is fiddly.
- Fun risk: might be more dashboard than game; needs juice.

## Done means
Running `revenant ./my-repo` produces a playable level where surviving mutants reach main, and the end screen lists at least one real file:line whose test I can strengthen to kill that mutant on the next run.
