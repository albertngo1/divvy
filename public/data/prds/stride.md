## Overview
Stride is a systems-programming puzzle game about data layout. Each level hands you a set of records and an access pattern; you arrange fields and iteration order to minimize cache misses and hit a target cycle budget. Think 'ECS vs OOP, the game.'

## Problem
Every performance-minded programmer has read the 'array-of-structs vs struct-of-arrays' and 'the physics of memory' posts and nodded — but few *feel* the cache. Stride turns the invisible cost of a cache miss into a tactile, scored puzzle: you watch cold rows fault in and hot rows stay resident, and you tune layout to beat par.

## How it works
- A level defines: records with typed fields (8/16/32/64-bit), a query ('for each Enemy, read position + health, write velocity'), and a machine model (line size 64B, L1 capacity, prefetcher stride depth, miss penalty).
- You drag fields between structs, split AoS into SoA, add padding, choose iteration order, and pick which arrays are 'hot.'
- Hit Run: a deterministic simulator streams the access trace through the modeled cache and animates line fills, evictions, and prefetch hits. It reports cycles vs par and misses vs par.
- Later levels add false sharing across two 'threads,' alignment traps, and a cold field that poisons a line if you leave it in the hot struct.
- Wordle-style: a daily layout challenge with a global leaderboard on cycle count.

## Technical approach
- Pure client-side, TypeScript + Canvas/WebGL; no backend for core play.
- Deterministic cache simulator: a set-associative model (configurable sets/ways/line size), LRU eviction, and a simple next-line/stride prefetcher. The access trace is generated by walking the level's query over the player's chosen layout, computing byte addresses from field offsets.
- Scoring = simulated cycles (hit=1, miss=penalty) + false-sharing penalty. Par is precomputed by an internal solver that brute-forces small layout permutations.
- Data model: Level { records, fields, query, machine, par }. Layout { structs[], fieldPlacement, iterationOrder }. Daily seed picks a level template + parameters.
- Hard part: making the cache model *simple enough to reason about but rich enough that SoA/padding/order genuinely matter*, and teaching it visually so players learn real intuition, not a toy's quirks.

## v1 scope
- 10 hand-authored levels escalating AoS→SoA→padding→iteration order.
- One deterministic cache+prefetcher sim with animated line fills.
- Cycle/miss score vs par, restart, shareable result string.

## Out of scope
- Real ISA/assembly, multi-level cache hierarchies, TLB.
- Multiplayer, accounts, persistence beyond localStorage.
- Level editor.

## Risks & unknowns
- Fun ceiling: could feel like homework instead of a game.
- Balancing so the 'obvious' SoA answer isn't always optimal (need traps that reward thought).
- Explaining the machine model without a wall of text.

## Done means
A player can complete 10 levels where at least 3 require non-obvious layout changes to beat par, the simulator produces identical cycle counts on repeat runs of the same layout, and a shareable daily result string encodes the day's score.
